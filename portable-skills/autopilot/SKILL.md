# /autopilot - Autonomous Development Agent

An autonomous development workflow that takes a Linear ticket and handles everything from branch creation to PR submission with minimal user intervention (one mandatory interaction: branch safety check).

**Prerequisite:** The ticket MUST have a detailed implementation plan (created via `/refine-ticket`). Autopilot executes the plan directly without modification or approval steps.

## Invocation

```
/autopilot TICKET-ID
```

**Note:** Ticket ID is required. If not provided, the skill will abort.

## Autonomy Model

**Mandatory User Interaction:** The **Pre-Flight Checklist** branch safety check always requires `AskUserQuestion`. You MUST complete this before any other work. After the branch safety check, the skill runs autonomously with no further approval checkpoints.

**After the pre-flight check:**
- File operations (write, edit, delete) proceed without prompts
- Test/lint/type failures are retried until resolved
- PR is created if any tasks complete successfully; partial-work PRs are allowed when some tasks are blocked (see Section 2.4)
- CI failures are retried until resolved

**Blocked task behavior:**
- The agent works persistently until each task is resolved
- All completed work (commits) is preserved on the branch

**Required permissions:** For full autonomy, ensure your `~/.claude/settings.json` includes the permissions listed in `.claude/skills/config.md` → Permissions section. These are generated by `/project-setup` based on your stack.

## Pre-Flight Checklist (MANDATORY — DO NOT SKIP)

**STOP. Complete the Pre-Flight Branch Safety Check from `.claude/skills/DEVELOPMENT.md` before doing ANYTHING else.** This MUST use AskUserQuestion — no auto-selection, even on main/staging.

Record: `WORKTREE_MODE = branch | worktree | remote-worktree`

## Stage 1: Context Gathering

### 1.1 Verify Ticket ID

If ticket ID was not provided as an argument, abort with error:

```
═══════════════════════════════════════════════════════════════
❌ Cannot proceed - No ticket ID provided
═══════════════════════════════════════════════════════════════
Usage: /autopilot TICKET-ID

Example: /autopilot TICKET-123
═══════════════════════════════════════════════════════════════
```

### 1.2 Fetch Linear Ticket Information

Use the Linear MCP tools to fetch the ticket:

```
mcp__linear-server__get_issue with id "TICKET-ID"
```

If the MCP tool fails, abort with error - autopilot cannot proceed without ticket context.

**Extract from the Linear issue:**
- `TICKET_SUMMARY`: Issue title
- `TICKET_TYPE`: Issue labels or type (bug, feature, chore)
- `TICKET_DESCRIPTION`: Issue description (markdown content)
- `ACCEPTANCE_CRITERIA`: Content under the `## Acceptance Criteria` header in the issue description (see `.claude/skills/acceptance-criteria/SKILL.md` for expected format — Given/When/Then, Requirements, Edge Cases, Out of Scope subsections)
- `IMPLEMENTATION_PLAN`: Content under the `## Implementation Plan` header in the issue description (see `.claude/skills/implementation-plan/SKILL.md` for expected format — Overview, Files to Modify, Implementation Tasks, Testing Strategy subsections)

### 1.3 Verify and Parse Implementation Plan (REQUIRED)

**CRITICAL:** Autopilot requires a detailed implementation plan to execute. The plan is assumed to be ready for direct execution without modification.

After fetching the ticket, check if the issue description contains a `## Implementation Plan` section with content. Plans are typically added by the `/refine-ticket` skill and follow the format defined in `.claude/skills/implementation-plan/SKILL.md`.

**If NO implementation plan is found - ABORT:**

```
═══════════════════════════════════════════════════════════════
❌ Cannot proceed - No implementation plan found
═══════════════════════════════════════════════════════════════
Ticket TICKET-ID does not have an implementation plan.

Please run:  /refine-ticket TICKET-ID

This will guide you through creating acceptance criteria and
a detailed implementation plan. Once complete, run /autopilot again.
═══════════════════════════════════════════════════════════════
```

**If an implementation plan IS found:**

The plan from `/refine-ticket` is assumed to be detailed and ready for execution. Parse it directly:

1. Extract from the `## Implementation Plan` section:
   - Overview/approach (from `### Overview`)
   - Files to modify with specific paths (from `### Files to Modify`)
   - Implementation tasks with specific steps (from `### Implementation Tasks`)
   - Testing strategy (from `### Testing Strategy`)
   - Any flagged questions or risks (from `### Risks and Mitigations`)
2. Determine execution mode (Stage 1.4)
3. Create the feature branch (Stage 1.5)
4. Save the parsed plan to `/tmp/TICKET-ID-plan.md` for state tracking
5. Proceed to dependency check (Section 1.4)

### 1.4 Check Ticket Dependencies (CRITICAL)

If the ticket description includes dependencies (e.g., "Dependencies: GRO-123, GRO-456"), **STOP and validate them** before proceeding:

1. **Fetch each dependency ticket** using `mcp__linear-server__get_issue`
2. **Check dependency status:**
   - Is it marked as Completed in Linear?
   - If not completed, check if it has both AC and implementation plan

**If ANY dependency is NOT completed AND lacks AC/plan - ABORT:**

```
═══════════════════════════════════════════════════════════════
❌ Cannot proceed - Unfinished dependencies detected
═══════════════════════════════════════════════════════════════
Ticket TICKET-ID depends on tickets that are not yet completed:

Dependencies blocking execution:
  ❌ GRO-789 (Status: Backlog) - No AC, No Plan
  ⚠️ GRO-456 (Status: In Progress) - Has AC ✓, No Plan ✗

Autopilot cannot proceed until blocking dependencies are:
1. Completed and merged, OR
2. Have well-defined AC and implementation plans

What would you like to do next?
═══════════════════════════════════════════════════════════════
```

Use AskUserQuestion with options:
```
Question: "This ticket has unfinished dependencies. What would you like to do?"
Header: "Dependencies"
Options:
  - Label: "Refine blocking ticket(s)", Description: "Use /refine-ticket to add AC and plans to dependencies"
  - Label: "Create new ticket", Description: "Use /linear-tickets to create a new unblocked ticket"
  - Label: "Exit autopilot", Description: "Stop and handle dependencies manually"
```

**Handle user choice:**
- **Refine blocking ticket(s)**: Show the user which ticket(s) need refinement, then exit autopilot
- **Create new ticket**: Exit autopilot and suggest running /linear-tickets
- **Exit autopilot**: Display final message and exit

**If ALL dependencies are either COMPLETED or have both AC and plan:**
- Display success message showing validated dependencies
- Proceed to Section 1.5 (Determine Execution Mode)

### 1.5 Determine Execution Mode

Parse `Depends On` fields from each task into a simple dependency graph (if present). Tasks without `Depends On` or with `Depends On: -` have no dependencies.

### 1.6 Create Feature Branch

Execute the branch creation for the chosen mode per `.claude/skills/DEVELOPMENT.md` "Branch creation by mode." Record `WORKTREE` in the plan file (`/tmp/TICKET-ID-plan.md`).

If the branch already exists, offer to use the existing branch or create a new one.

## Stage 2: Implementation

### 2.1 Begin Implementation

Update the plan file:
```markdown
## Status: IN_PROGRESS
```

Proceed to Section 2.2 (TDD Loop) to implement each task sequentially.

### 2.2 TDD Loop

Implement each task in the plan sequentially, following the task dependencies defined in `Depends On` fields. Tasks without dependencies or with `Depends On: -` can be implemented in any order.

**TDD cycle:** Write/update tests → run tests (expect failure) → implement minimal code → run tests (expect pass). If tests fail after implementation, analyze the failure, fix, and re-run. Try different approaches if the same fix isn't working — re-read related code, reconsider strategy, or refactor.

**All file operations use `$WORKTREE/path/to/file`.** Run tests from the worktree using the commands from `.claude/skills/tech-stack.md` → Commands Quick Reference:
```bash
cd $WORKTREE && <test command from tech-stack.md for this repo>
```

**Before committing:** Run the lint and type-check commands from tech-stack.md. Only proceed after ALL checks pass.

**Commit format:**
```bash
git -C $WORKTREE add <specific_files>
git -C $WORKTREE commit -m "$(cat <<'EOF'
TICKET-ID Task description

- What was changed
- Why it was changed
EOF
)"
```

- Start with ticket ID, imperative mood, first line under 72 chars
- Stage specific files, never `git -C $WORKTREE add -A`

**Update plan file** after each task: mark status (`✅ COMPLETED` / `⚠️ BLOCKED`) and record commit hash.

### 2.3 Blocked Task Report

**If any task is blocked, display this report. If ALL remaining tasks are blocked/skipped, exit (do not proceed to Stage 3). If some tasks completed, proceed to Stage 3.**

```
═══════════════════════════════════════════════════════════════
❌ Autopilot Stopped - Tasks Blocked
═══════════════════════════════════════════════════════════════
Ticket: TICKET-ID
Branch: branch_name

Tasks Completed: X/Y
Tasks Blocked: B
Tasks Skipped: S (due to blocked dependencies)

───────────────────────────────────────────────────────────────
⚠️ Blocked Tasks:

Task N: [Title]
  Reason: [Detailed failure message]
  Files: [affected files]
  Last Error:
    [Test output or error message]

⏭️ Skipped Tasks (blocked dependencies):

Task M: [Title]
  Depends on blocked task: N
───────────────────────────────────────────────────────────────

Completed Work:
  • Commits on branch: N
  • Files modified: [list]

Next Steps:
  1. Review the errors above
  2. Fix the issues manually or update the implementation plan
  3. Run /autopilot TICKET-ID again to retry

Branch preserved at: origin/branch_name
═══════════════════════════════════════════════════════════════
```

**Exit the skill after displaying this report if no tasks completed.** If some tasks completed, proceed to Stage 3 to create a PR with partial work.

## Stage 3: Finalization

### Completion Gate (READ THIS BEFORE STARTING STAGE 3)

**You MUST complete ALL of the following before reporting completion to the user:**

- [ ] 3.1: Verify all changes are committed and pushed
- [ ] 3.2: PR created with gh command
- [ ] 3.3: Final report displayed
- [ ] 3.4: Cleanup performed

**If any of these are incomplete, go back and complete them now.**

### 3.1 Verify and Push

Verify all changes are committed and pushed to remote:

```bash
# Check for uncommitted changes
git status

# If there are uncommitted changes (like Gemfile.lock), commit them
git add <files>
git commit -m "TICKET-ID <description>"

# Push to remote
git push
```

### 3.2 Create PR with gh

Create a draft pull request using the `gh` command:

```bash
gh pr create --draft \
  --title "TICKET-ID: <summary>" \
  --body "$(cat <<'EOF'
## Summary

<Brief description of changes>

### Changes Made

- ✅ <Change 1>
- ✅ <Change 2>

### Files Modified

- <file 1>
- <file 2>

### Test Results

```
<Test output summary>
```

### Configuration Required (if applicable)

<Any env vars or config needed>

---
EOF
)"
```

The `gh pr create` command returns the PR URL (e.g., `https://github.com/org/repo/pull/123`).

**Extract the PR number from the URL for the final report.**

### 3.3 Display Final Report

Display the completion summary to the user:

```
═══════════════════════════════════════════════════════════════
✅ Autopilot Complete - TICKET-ID
═══════════════════════════════════════════════════════════════
Ticket: <ticket title>
Branch: <branch name>
PR: <PR URL>

Tasks Completed: X/Y
- ✅ Task 1: <description> (<commit hash>)
- ✅ Task 2: <description> (<commit hash>)
[If any blocked/skipped:]
- ⚠️ Task N: <description> - BLOCKED: <reason>
- ⏭️ Task M: <description> - SKIPPED: depends on Task N

Commits: N
Files Created/Modified: N
Test Coverage: X examples, Y failures

Next Steps:
1. Review PR and approve
2. [Any specific instructions, e.g., add env vars]
3. Merge when ready
═══════════════════════════════════════════════════════════════
```

### 3.4 Cleanup

**Always clean up the plan file:**
```bash
rm -f /tmp/TICKET-ID-plan.md
```

**If WORKTREE_MODE is `worktree` or `remote-worktree`**, also clean up the worktree:
```bash
git worktree remove /tmp/autopilot-TICKET-ID
```
